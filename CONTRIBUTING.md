# Contributing Guidelines

Thank you for your interest in contributing to the GitHub Copilot CLI documentation repository! We welcome contributions that improve the documentation, fix errors, add examples, or enhance the user experience.

## Getting Started

### Prerequisites

- A GitHub account
- Git installed locally
- Basic knowledge of Markdown formatting
- Familiarity with GitHub Pull Request workflow

### Setting Up Locally

1. **Fork the repository** on GitHub
2. **Clone your fork**:
   ```bash
   git clone https://github.com/YOUR-USERNAME/baddieai.git
   cd baddieai
   ```
3. **Create a branch** for your changes:
   ```bash
   git checkout -b feature/your-feature-name
   ```

## Making Changes

### Documentation Style

- Use clear, concise language
- Write in a professional but approachable tone
- Include practical examples where helpful
- Use proper Markdown syntax
- Test all command examples before submitting
- Verify all links are functional

### File Organization

- **README.md**: Main documentation for users
- **REPOSITORY-MIGRATION.md**: Migration and setup guides
- **.github/workflows/**: GitHub Actions workflow examples
- **CONTRIBUTING.md**: This fileâ€”contribution guidelines
- **SECURITY.md**: Security policy and reporting guidelines

### Markdown Best Practices

- Use descriptive headers (`##`, `###`)
- Format code with triple backticks and language tags
- Use lists for sequential steps or grouped items
- Include links to official documentation when referencing features
- Add horizontal rules (`---`) to separate major sections

## Workflow

### Submitting Changes

1. **Make your changes** in your feature branch
2. **Commit your changes**:
   ```bash
   git add .
   git commit -m "Brief description of changes"
   ```
3. **Push to your fork**:
   ```bash
   git push origin feature/your-feature-name
   ```
4. **Open a Pull Request** on GitHub

### Commit Message Guidelines

- Keep the first line under 72 characters
- Use present tense ("Add feature" not "Added feature")
- Be descriptive but concise
- Reference issues when applicable (e.g., "Fixes #123")

Examples:
- `Add example for PAT authentication`
- `Fix broken link in README`
- `Update migration guide with latest GitHub features`
- `Improve workflow examples with better comments`

## Pull Request Process

1. **Ensure all changes are documented** in your PR description
2. **Verify that links work** and examples are accurate
3. **Check for typos** and formatting issues
4. **Wait for review** from maintainers
5. **Address feedback** if requested
6. **Celebrate** when your PR is merged! ðŸŽ‰

### What to Include in Your PR

- **Title**: Clear, descriptive summary of changes
- **Description**: 
  - What changed and why
  - Any related issues or context
  - How you tested the changes (e.g., verified links, tested commands)
- **Screenshots**: If applicable (e.g., showing formatting improvements)

## Types of Contributions

We welcome:

- **Documentation improvements**: Clarifying existing content
- **Error fixes**: Correcting typos, broken links, or outdated information
- **New examples**: Adding helpful code examples or use cases
- **Migration guides**: Updating or expanding setup instructions
- **Workflow enhancements**: Improving GitHub Actions examples
- **Accessibility improvements**: Making documentation easier to read and understand

## Code of Conduct

### Our Standards

- Be respectful and inclusive
- Welcome newcomers and help them learn
- Focus on constructive feedback
- Respect differing viewpoints and experiences
- Accept responsibility and apologize for mistakes

### Unacceptable Behavior

- Harassment, trolling, or insulting comments
- Publishing others' private information
- Any conduct that would be inappropriate in a professional setting

## Questions or Issues?

- **General questions**: Open a [Discussion](https://github.com/hustlenhealai-ui/baddieai/discussions)
- **Bug reports**: Open an [Issue](https://github.com/hustlenhealai-ui/baddieai/issues)
- **Documentation unclear?**: Suggest improvements via a Pull Request
